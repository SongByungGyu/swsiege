# 서머너즈워 점령전 트래커 — 디자인 문서

**작성일**: 2026-05-13
**목적**: 길드 내부용 점령전 기록 자동 누적 및 방어덱 기반 공격 통계 분석 도구

---

## 1. 개요

### 1.1 해결하려는 문제

길드 점령전에서 가장 중요한 의사결정은 "이 방어덱을 어떤 공격덱으로 깰 것인가"다. 현재는 길드원들의 머릿속/디스코드 메모/스프레드시트에 흩어져 있어, 과거 우리 길드가 같은 방어덱에 어떤 공격을 시도했고 결과가 어땠는지를 체계적으로 활용하기 어렵다.

### 1.2 핵심 가치

- **방어덱 기준 검색**: 특정 몬스터(예: "여몽")가 포함된 방어덱들과, 그에 대응한 우리 길드의 공격 기록을 한눈에 본다.
- **자동 누적**: 길드원들이 SWEX(SW Exporter)를 사용 중이라면, 점령전 화면을 여는 것만으로 데이터가 자동으로 쌓인다.
- **승률 데이터**: 공격덱별 시도 횟수와 승률을 표시해, 다음 점령전 전략 수립의 근거로 삼는다.

### 1.3 범위 (Scope)

- ✅ 단일 길드 내부 도구 (멀티 길드/SaaS 아님)
- ✅ 길드원 약 20~30명 규모
- ✅ 웹 기반 (PC + 모바일 반응형)
- ❌ 실시간 게임 자동 로그인 / 공식 API 사용 (Com2uS 미제공)
- ❌ RTA, 길드전, 일반 콘텐츠 분석 (점령전에 집중)

## 2. 시스템 아키텍처

세 컴포넌트로 구성:

```
┌─────────────────────────┐         ┌──────────────────────┐         ┌────────────────────┐
│ 길드원 PC               │         │ Supabase (Backend)   │         │ Next.js Web App    │
│ ┌─────────────────────┐ │  HTTPS  │                      │         │                    │
│ │ SWEX                │ │  POST   │  - Postgres DB       │  HTTPS  │  - 방어덱 검색      │
│ │ + Siege Tracker     │─┼────────▶│  - Auth (이메일)     │◀────────│  - 통계 시각화      │
│ │   Plugin (우리 제작)│ │  JSON   │  - REST API (자동)   │         │  - 정렬/거점 필터   │
│ └─────────────────────┘ │         │  - Realtime (옵션)   │         │  - 길드원 관리      │
│  ↑ 게임 패킷 캡처       │         └──────────────────────┘         └────────────────────┘
│  (SWEX가 MITM 프록시)   │
└─────────────────────────┘
```

### 2.1 컴포넌트별 역할

**SWEX 플러그인** (`siege-tracker-plugin`)
- SWEX는 게임이 서버와 주고받는 HTTPS 패킷을 가로채는 일렉트론 앱(오픈소스).
- 우리 플러그인은 그중 **점령전 관련 명령(command)** 만 골라서 Supabase로 POST한다.
- 길드원이 게임에서 점령전 관련 화면을 열 때마다 데이터가 자동 전송됨.

**Supabase 백엔드**
- Postgres DB로 모든 점령전 기록 저장.
- 인증은 공유 비밀번호 방식이므로 Supabase Auth는 사용하지 않음. 대신 Next.js API 라우트에서 비밀번호 검증 + 세션 쿠키 발급.
- DB 접근은 서버 측(서비스 키)으로만, 클라이언트는 우리 API를 통해서만 접근.
- REST/GraphQL 자동 API는 비활성화하여 Supabase를 단순 DB로 사용.

**Next.js 웹**
- 길드원이 로그인 후 방어덱 검색, 통계 조회.
- 정적 호스팅 + 서버리스 API 라우트로 Vercel 무료 배포 가능.

## 3. 데이터 모델

### 3.1 핵심 엔티티

```
guild (우리 길드 — 1개만 존재)
  id, name, world_id

enemy_guild (상대 길드)
  id, name, world_id, last_seen_at

defense_deck (방어덱 1세트 = 3마리)
  id, enemy_guild_id, slot_index (점령전 거점 위치),
  monster_1_id, monster_2_id, monster_3_id,
  artifacts_summary (각 몬스터의 핵심 아티팩트 — 옵션),
  captured_at, captured_by_in_game_name (게임 내 닉네임)

attack_record (우리 길드원이 그 방어덱을 공격한 기록)
  id, defense_deck_id, attacker_in_game_name (게임 내 닉네임),
  initial_monster_1_id, initial_monster_2_id, initial_monster_3_id,
  replacement_monsters (셔플로 투입된 몬스터들 — JSON 배열),
  result (win|lose),
  points_earned,
  attacked_at

(개인 user 계정 없음 — 공유 비밀번호 방식. 공격자/캡처자는
 SWEX가 보내준 게임 내 닉네임으로 식별.)

monster (몬스터 마스터 데이터 — SW 전체 몬스터 정보)
  id, name_ko, name_en, element, archetype, awakened, image_url
  (2★~5★ + 미각성/각성/2차각성 모두 포함)

(시즌 개념 없음 — 모든 기록은 시간순으로 누적. captured_at/attacked_at으로
 시간 기준 식별. 같은 길드의 과거 방어덱과 현재 방어덱이 같은 검색 결과에
 섞일 수 있다는 점을 의도적으로 수용.)
```

### 3.2 검색 동작

**방어덱 검색 (부분 매칭)**:
1. 사용자가 검색창에 몬스터 1~3마리 입력 (자동완성)
2. 입력한 모든 몬스터를 포함하는 `defense_deck` 조회
3. 각 방어덱마다 연결된 `attack_record`를 집계 → 공격덱별 시도/승률

**예시 SQL 개념**:
```sql
SELECT d.*, COUNT(*) as appearance_count
FROM defense_deck d
WHERE d.monster_1_id IN (?, ?, ?)
   OR d.monster_2_id IN (?, ?, ?)
   OR d.monster_3_id IN (?, ?, ?)
GROUP BY d.id
ORDER BY appearance_count DESC
```

## 4. SWEX 플러그인

### 4.1 기본 구조

SWEX 플러그인은 `proxy.on('apiCommand', handler)` 콜백을 등록해서 게임 명령마다 호출됨. 우리 플러그인은 점령전 관련 명령만 필터링.

```javascript
// siege-tracker-plugin.js (의사 코드)
module.exports = {
  defaultConfig: { enabled: true, apiUrl: 'https://our-app.vercel.app/api/ingest', authToken: '' },

  init(proxy, config) {
    proxy.on('BattleGuildSiegeGetInfo', (req, resp) => {
      this.uploadSiegeInfo(resp, config);
    });
    proxy.on('BattleGuildSiegeStart', (req, resp) => { /* 공격 시작 기록 */ });
    proxy.on('BattleGuildSiegeResult', (req, resp) => { /* 공격 결과 기록 */ });
    // 정확한 명령 이름은 실제 SWEX 캡처 + SWGT 플러그인 참고로 확정
  }
};
```

### 4.2 캡처 대상 명령 (구현 시점에 확정)

기존 오픈소스 도구(SWARFARM, SWGT 플러그인)의 코드를 참고해서 점령전 관련 SW API 커맨드 이름을 확정한다. 후보:
- 점령전 정보 조회 (다른 길드 정찰)
- 점령전 공격 시작
- 점령전 공격 결과
- 길드 점령전 진행 상태

### 4.3 보안

- 플러그인에 길드별 발급된 `authToken`을 설정. 우리 API에서 토큰 검증.
- 토큰은 길드 관리자가 발급 → 길드원에게 1:1 전달.

### 4.4 배포

- GitHub Releases로 `.js` 파일 배포.
- 길드원이 SWEX 폴더의 `plugins/`에 넣고, 설정에 본인 토큰 입력.

## 5. 프론트엔드 화면

### 5.1 페이지 구성

1. **로그인** — 공유 비밀번호 입력 (한 번 통과하면 세션 유지)
2. **닉네임 선택** — 사이트 첫 진입 시, 길드원 목록(DB에서 자동 추출)에서 본인 게임 닉네임 선택 → 브라우저 localStorage에 저장. "내 기록" 등에 활용.
3. **메인 (방어덱 검색)**
   - 검색창: 몬스터 자동완성 입력 (1~3개)
   - 필터: 거점 위치, 정렬(만남 횟수/승률/최근순)
   - 결과 카드: 방어덱(3마리 이미지) + 만남 횟수 + 공격 기록 펼치기
   - 같은 길드의 옛 방어덱과 새 방어덱이 함께 보이므로 `captured_at`을 명시
4. **공격 기록 상세** — 누가 언제 어떤 덱으로 공격했고 결과는?
5. **내 기록** — 선택된 본인 닉네임으로 필터링한 출전 기록 모음
6. **설정** — 공유 비밀번호 변경, SWEX 플러그인 토큰 확인
7. **플러그인 안내** — SWEX 설치 가이드 + 우리 플러그인 다운로드 + 입력해야 할 공유 비밀번호(=토큰) 안내

### 5.2 UX 메모

- 모바일에서도 점령전 중에 빠르게 검색할 수 있어야 함 → 모바일 반응형 우선.
- 방어덱 카드는 몬스터 이미지를 크게(검색 정확도 빠른 시각 확인).
- 검색 결과는 페이지네이션보다 무한 스크롤이 점령전 중 사용성에 적합.

## 6. 설계 결정사항 (기본값 제안)

브레인스토밍 단계에서 미루었던 항목들에 대한 기본값 제안. 검토 후 조정 가능.

### 6.1 시즌 처리 (제거)

- **시즌 개념 폐기**. 모든 방어덱·공격 기록은 시간순으로 무한 누적.
- 같은 상대 길드의 과거 방어덱과 현재 방어덱이 같은 검색 결과에 함께 노출됨을 의도적으로 수용.
- 시간 식별은 `captured_at` / `attacked_at` 타임스탬프로 충분.
- 추후 데이터가 과다해지면 (예: 1년치 누적 후) "최근 N개월 필터" 같은 시간 필터를 도입할 수 있음. Phase 2에서는 도입하지 않음.

### 6.2 정찰 데이터 포함

- **포함**. 우리가 공격 안 한 방어덱도 DB에 저장.
- 검색 결과에 "0회 공격 (정찰만)" 으로 표시 → 미답 덱 식별 가능.

### 6.3 인증

- **공유 비밀번호 1개 방식**.
- 사이트 접속 시 비밀번호 입력 → 통과하면 세션 쿠키로 일정 기간 유지.
- 길드원은 길드 마스터/관리자로부터 비밀번호를 받아서 사용.
- 개인 계정/이메일/회원가입 없음.
- **공격자/캡처자 식별**: SWEX가 보내주는 게임 내 닉네임으로 자동 식별. 웹에서 직접 입력하는 항목이 아니므로 위조 불가.
- **SWEX 플러그인 토큰**: 사이트 비밀번호와 **동일한 값을 사용**. 비밀번호 1개로 사이트 접속과 플러그인 데이터 전송 인증을 모두 처리. 환경변수로 관리.
- **트레이드오프**:
  - 비밀번호 유출 시 외부인 접속 가능 → 정기 변경 권장 (설정 페이지에서 변경 가능).
  - 누가 사이트에 접속했는지 감사(audit) 불가 → 길드 내부용 수준에서 허용.
  - "관리자 권한"이 없음 → 모두 동등. 비밀번호 변경 같은 민감 작업은 관습적으로 길드 마스터가 수행.

### 6.4 공격 셔플 처리

- 점령전 공격은 초기 3마리 + 셔플(replacement) 2번 = 최대 5마리.
- 모든 사용 몬스터 기록 (DB에 `initial_monsters` + `replacement_monsters` 둘 다).
- 통계 집계는 **초기 3마리 기준** (셔플은 결과에 미치는 영향이 작고, 동일 초기덱을 묶어야 의미 있는 표본).
- 상세 화면에서 셔플 이력 확인 가능.

### 6.5 데이터 보관 기간

- 모든 기록 영구 보관 (길드 규모상 DB 용량 부담 없음).
- 백업: Supabase 자동 일일 백업 (무료 티어 7일 보관).

## 7. 단계별 구현 순서 (개요)

상세 구현 계획은 다음 단계의 implementation plan에서 작성. 큰 흐름은:

1. **Phase 1: 기반 셋업**
   - Next.js 프로젝트 생성, Supabase 프로젝트 생성, DB 스키마 작성
   - 공유 비밀번호 인증 (Next.js API 라우트 + 세션 쿠키)
2. **Phase 2: 데이터 모델 + 수동 입력**
   - 방어덱/공격기록을 수동으로 입력하는 폼 (플러그인 작동 전 검증용)
   - 방어덱 검색 UI 구현
3. **Phase 3: SWEX 플러그인**
   - 점령전 API 커맨드 조사 (SWGT 코드 참고 + 실제 패킷 캡처)
   - 플러그인 작성 + 토큰 인증
   - 자동 업로드 동작 검증
4. **Phase 4: 통계 + 시각화**
   - 승률, 시도 횟수, 트렌드 그래프
   - 시간순 추이 (월별/주별)
5. **Phase 5: 배포 + 길드원 온보딩**
   - Vercel 배포, 도메인 연결
   - 길드원용 설치 가이드 작성

## 8. 위험 요소 및 대응

| 위험 | 대응 |
|---|---|
| **점령전 API 커맨드 분석 난이도** | SWGT 등 기존 오픈소스 도구 참고. Phase 3 초반에 PoC 먼저. |
| **Com2uS 약관 위반 가능성** | SWEX는 회색 지대지만 현재까지 광범위 사용 + 묵인. 내부 도구로만 운용, 공개 배포 없음. |
| **길드원의 SWEX 사용률 저조** | 핵심 인원(주공·서브) 5~10명만 깔아도 데이터 누적은 충분. 수동 입력 폼도 병행. |
| **시즌 종료 후 방어덱 변경** | 같은 길드의 옛/새 방어덱이 한 검색 결과에 섞임. `captured_at` 정렬과 "최근 N개월" 필터(추후)로 식별. |
| **Supabase 무료 티어 한계** | 길드 규모상 무료 티어로 수년 운영 가능. 한계 도달 시 유료 전환($25/월). |
